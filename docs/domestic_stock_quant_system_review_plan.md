# 国内股票量化系统方案评审与二次落地计划

版本：v0.1  
日期：2026-05-17  
基于文档：`docs/domestic_stock_quant_system.md`

## 1. 评审结论

原方案的总体方向是合理的：它没有把系统绑定到单一开源框架，而是采用“数据与研究中台 + 回测/执行适配层 + 独立风控”的方式，适合国内股票量化这种规则复杂、券商接口不统一、数据口径敏感的场景。

但原方案仍偏“完整蓝图”，对早期落地来说有几处过宽：

- 同时提到 Qlib、vectorbt、vn.py、Backtrader、LEAN、NautilusTrader，容易造成技术栈摇摆。
- 对国内股票数据质量、财务数据可得日、复权口径、指数成分历史的难度说得还不够重。
- “自研事件驱动回测”方向正确，但工作量和验收边界需要更清晰。
- “纸面交易、小资金实盘”写得合理，但券商接口、权限、合规和人工兜底要求需要前置。
- 阶段周期偏乐观，尤其是事件回测和实盘适配。

二次收敛后的建议是：

> 第一阶段只做“日频多因子研究 + 可复现数据 + 保守回测”。  
> 第二阶段做“A 股规则事件回测 + 纸面交易”。  
> 第三阶段在明确券商接口后，再做小资金实盘。

## 2. 不同角色视角评审

### 2.1 资深从业人员视角

关心点：

- 这套系统能不能避免回测幻觉。
- 数据是否可信。
- 风控是否独立。
- 实盘链路是否可控。
- 策略上线是否有审查和复盘机制。

正向评价：

- 将策略、组合、风控、执行拆开是正确的。
- 强调 T+1、涨跌停、停牌、手续费、复权，是 A 股系统必须做的基本功。
- 先纸面交易再小资金实盘是正确路径。
- 数据版本、实验追踪、盘后复盘这些设计有长期价值。

主要质疑：

- 免费数据源不应被当成生产数据基础。AkShare、BaoStock 适合开发验证，但稳定性、历史修正、复权口径、财务字段完整性都需要二次校验。
- 因子研究如果没有严格的可得日和样本外验证，很容易得到虚假收益。
- “自研事件驱动回测 4-6 周”偏乐观。如果要覆盖完整 A 股公司行为、分红送转、停牌、涨跌停、交易费用、卖出零股、撤单等细节，周期会显著增加。
- 股票实盘不是单纯技术问题，券商接口权限、交易终端限制、自动化合规边界、账户风控都需要提前确认。

资深从业人员会建议：

- 先不要碰分钟级和实盘。
- 先用日频把数据、因子、回测、报告闭环打磨扎实。
- 用保守成交假设，宁愿低估收益，不要高估。
- 所有策略必须有“不可成交分析”和“换手/容量分析”。

### 2.2 门外汉小白视角

关心点：

- 这东西到底先做什么。
- 哪些工具必须学。
- 怎么知道做出来的结果不是骗人的。
- 什么时候能看到成果。

正向评价：

- 原文覆盖了完整流程，可以让新人知道量化不只是写一个买卖策略。
- 对 A 股规则有提醒，能避免“拿美股/币圈思路直接套 A 股”的误区。

主要困惑：

- 工具名太多，不知道先学哪个。
- “Qlib + vectorbt + 自研回测 + vn.py”听起来都要学，实际会劝退。
- “因子评价、组合构建、风险预算、OMS/EMS”这些词需要更明确的入门顺序。

给小白的二次解释：

第一版只需要理解四件事：

1. 数据是否干净：今天能不能拿到过去每天每只股票的真实价格、停牌、涨跌停和复权信息。
2. 信号是否有效：用历史数据看某些特征是否能预测未来收益。
3. 回测是否保守：买不到、卖不出、手续费、T+1 都要算进去。
4. 实盘是否可控：先模拟，不直接拿真金白银下单。

小白不应该第一天就学完所有框架。建议学习顺序：

1. Python、pandas、基本金融数据。
2. A 股交易规则。
3. 因子评价和简单回测。
4. 组合构建和风控。
5. 纸面交易和实盘接口。

### 2.3 实际工程师视角

关心点：

- 模块边界是否清晰。
- 数据结构是否稳定。
- 哪些依赖会拖慢交付。
- 有没有可测试、可监控、可部署的路径。

正向评价：

- 使用 Python、Parquet、DuckDB、PostgreSQL 是务实的。
- 先 Docker Compose 后 Kubernetes 是合理的。
- 用 Pydantic 做配置和领域模型校验是可行的。
- 把券商接口抽象成 `BrokerAdapter` 是必要的。

主要质疑：

- 原项目结构偏完整，但第一版不应创建太多空模块。
- 同时接 Qlib、vectorbt、MLflow、Prefect、Grafana 会让 MVP 变重。
- 自研事件回测如果没有先定义最小撮合规则，很容易做成长期工程。
- 文档没有明确测试策略，例如数据校验测试、回测基准测试、风控规则测试。

工程师会建议：

- 第一版只选 `Parquet + DuckDB + pandas/polars + Pydantic + pytest`。
- vectorbt 可以作为可选依赖，不要成为核心数据结构。
- Qlib 放到第二阶段，等数据和回测稳定后再接。
- MLflow 可以先只记录本地实验，不急着搭服务。
- 不要一开始搭 Prometheus/Grafana，先把日志、日报、错误告警做好。

### 2.4 量化分析员视角

关心点：

- 能否快速研究因子。
- 能否避免未来函数。
- 能否做横截面分析、分层收益、IC/IR、换手、行业暴露。
- 回测报告是否能解释收益来源。

正向评价：

- 原方案将因子处理、因子评价、组合构建独立出来是科学的。
- 对股票池、ST、停牌、上市不足 N 日、低流动性过滤的设计合理。
- 先做沪深 300/中证 500/中证 1000 这样的指数池，比直接全 A 更稳。

主要质疑：

- 因子标签和财务数据可得日没有被提升到最高优先级。
- 没有明确训练集、验证集、测试集和滚动样本外机制。
- 没有明确基准，例如沪深 300、中证 500、中证 1000、等权指数或行业中性基准。
- 多因子组合方式还偏粗，需要区分打分合成、回归预测、排序模型和优化器。

量化分析员会建议：

- 第一版不要追求 ML 模型，先做透明的规则因子。
- 每个因子必须有单因子评价报告。
- 每个策略必须有基准对比、换手、行业暴露、风格暴露。
- 先做周频/月频调仓，降低交易成本和不可成交影响。

### 2.5 数据工程师视角

关心点：

- 数据是否可追溯。
- 是否有数据质量检查。
- 增量更新能不能稳定运行。
- 数据字段和口径是否固定。

正向评价：

- `raw/clean/feature` 三层数据设计正确。
- 使用 Parquet 分区适合离线研究。
- `source/as_of_date/ingested_at/version` 这些字段非常重要。

主要质疑：

- 原文没有给出明确的数据质量规则。
- 免费数据源字段可能变动，必须做 schema 校验。
- 多数据源对账和冲突处理没有说明。
- 复权因子、涨跌停价格、停牌数据必须被当成一等数据，不是附属字段。

数据工程师会建议：

- 第一版先固定一个主数据源，避免多源混乱。
- 用第二数据源做抽样校验，而不是并行混用。
- 建立数据质量报告，例如缺失率、重复率、交易日覆盖率、异常价格、成交量异常。

### 2.6 风控/运营视角

关心点：

- 系统失误时能否停下来。
- 订单是否会重复、超额、越权。
- 账户和系统持仓是否一致。
- 人是否能理解和接管。

正向评价：

- 原文把风控独立出来是正确的。
- kill switch、一键撤单、对账、异常行情暂停都提到了。

主要质疑：

- 风控规则需要分为“研究期风控”和“实盘期风控”，否则早期实现压力过大。
- 自动交易的审批、人工确认、日志留痕没有具体化。
- 小资金实盘前的准入条件要量化。

风控/运营会建议：

- 纸面交易至少连续稳定 20 个交易日。
- 每日盘前必须有数据更新检查和账户检查。
- 每日盘后必须有成交、持仓、资金对账。
- 实盘初期只允许白名单策略、白名单股票池、固定最大资金。

## 3. 原文中偏模糊或不可靠的地方

### 3.1 技术栈范围过宽

原文同时列出多个框架，容易给人一种“这些都要用”的感觉。实际应该按阶段选择：

| 阶段 | 应使用 | 暂不使用 |
|---|---|---|
| MVP | pandas/polars、Parquet、DuckDB、Pydantic、pytest | Qlib、vn.py、ClickHouse、Kubernetes |
| 因子研究增强 | Qlib、MLflow | 实盘框架 |
| 高速回测 | vectorbt | 深度 OMS/EMS |
| 纸面交易 | 自研 PaperBroker、日志、日报 | 真正自动下单 |
| 实盘 | 券商适配器、OMS/EMS、风控 | 未验证策略自动上线 |

二次结论：

- 第一版不要让 Qlib、vn.py、vectorbt 成为系统核心。
- 第一版核心是内部数据结构和 A 股规则。
- 开源框架作为插件和参考，而不是主干。

### 3.2 数据源假设偏乐观

模糊点：

- AkShare、BaoStock、Tushare Pro 的定位没有区分开发、研究、生产。
- 没有明确商业数据源何时引入。
- 没有指定主数据源和备份校验源。

二次结论：

- 开发期可以用免费源。
- 研究期至少要做多源抽样对账。
- 实盘前必须确定稳定数据源，最好使用商业数据或券商/数据商数据。

### 3.3 自研事件回测工作量偏低估

模糊点：

- “自研轻量引擎”没有定义轻量到什么程度。
- A 股撮合规则只写了方向，没有拆成可验收功能。

二次结论：

第一版事件回测只支持：

- 日频 bar。
- 收盘后生成信号。
- 次日开盘价或 VWAP 成交。
- 停牌不可交易。
- 涨停不可买，跌停不可卖。
- 买入 100 股整数倍。
- T+1 卖出限制。
- 佣金、印花税、过户费、滑点。

暂不支持：

- Tick 撮合。
- 盘口队列。
- 部分成交。
- 复杂订单类型。
- 公司行为现金流精细模拟。

### 3.4 实盘部分过早

模糊点：

- QMT、PTrade、券商 API 的权限、稳定性、可用功能差异很大。
- 文档没有明确“什么时候允许实盘”。

二次结论：

实盘不是第三步，而是纸面交易稳定后的独立项目。准入条件：

- 数据任务连续 20 个交易日无重大缺失。
- 纸面交易连续 20 个交易日稳定。
- 每笔订单有完整日志。
- 风控能拦截超限订单。
- 盘前、盘中、盘后检查表可执行。
- 手动暂停、撤单、恢复流程通过演练。

### 3.5 Qlib 和 ML 模型引入偏早

模糊点：

- 原文把 Qlib 放在因子研究推荐中，但没有说明它不是第一天必须用。
- 新手容易把系统复杂度花在模型上，而不是数据和回测上。

二次结论：

- 前 1-2 个月不建议引入复杂 ML。
- 先做透明因子：动量、反转、波动率、流动性、市值、估值。
- 等单因子评价、组合回测、滚动样本外稳定后，再引入 Qlib/LightGBM。

### 3.6 阶段时间需要更现实

原文阶段时间对成熟团队可能可行，但对从零搭建的小团队偏乐观。

修正建议：

| 阶段 | 原估计 | 修正估计 |
|---|---:|---:|
| 项目初始化 | 1 周 | 1-2 周 |
| 日频数据与基础研究 | 2-4 周 | 4-6 周 |
| 向量化回测 MVP | 2-3 周 | 3-4 周 |
| 事件驱动回测 | 4-6 周 | 6-10 周 |
| 纸面交易 | 4-8 周 | 8-12 周 |
| 小资金实盘 | 1-3 个月 | 取决于券商接口，至少 2-3 个月 |

## 4. 原文中比较可行和科学的内容

### 4.1 架构分层是正确的

数据、研究、策略、组合、回测、执行、风控、复盘分层合理。尤其是策略不直接下单，而是输出信号或目标仓位，这是专业系统应该有的边界。

### 4.2 先研究、再回测、再纸面、再实盘是正确路径

这能显著降低真实资金风险，也能帮助发现回测和现实之间的偏差。

### 4.3 A 股规则被纳入核心设计

T+1、涨跌停、停牌、ST、交易单位、印花税、复权这些规则如果不处理，回测结果基本不可信。原文把它们放进核心设计是正确的。

### 4.4 数据版本和实验追踪有长期价值

量化系统最怕“同一个策略今天跑和明天跑结果不一样但不知道为什么”。数据版本、参数版本、代码版本、实验记录是可复现的基础。

### 4.5 MVP 选择日频多因子是合理的

日频多因子比分钟级、高频、盘口策略更适合第一阶段。它的数据压力低、解释性强、交易链路更简单，适合作为系统第一条闭环。

### 4.6 独立风控是必须的

风控不能只写在策略里。原文的订单风控、账户风控、组合风控、市场风控方向正确。

## 5. 二次收敛后的技术选型

### 5.1 第一阶段固定选型

| 模块 | 最终选择 | 原因 |
|---|---|---|
| 语言 | Python 3.11+ | 生态成熟 |
| 项目管理 | uv | 快速、依赖锁定清晰 |
| 配置 | Pydantic + YAML/TOML | 配置校验强 |
| 数据存储 | Parquet + DuckDB | 单机研究足够，成本低 |
| 元数据 | SQLite 起步，后续 PostgreSQL | MVP 阶段不急着引入数据库运维 |
| 数据处理 | pandas + polars + pyarrow | 兼顾生态和性能 |
| 回测 | 自研日频回测核心 | 先保证 A 股规则可控 |
| 参数扫描 | 暂缓 vectorbt，后续作为辅助 | 避免早期抽象过多 |
| 因子研究 | 自研基础因子评价 | Qlib 后置 |
| 实验记录 | 本地 Markdown/JSON 报告，后续 MLflow | 降低 MVP 运维复杂度 |
| 调度 | 手动 CLI + cron，后续 Prefect | 先保证任务可重复 |
| 监控 | 日志 + 运行报告，后续 Grafana | 实盘前再升级 |

### 5.2 第二阶段候选选型

| 模块 | 候选 | 引入条件 |
|---|---|---|
| Qlib | 因子/模型研究 | 基础数据和回测稳定后 |
| vectorbt | 高速参数扫描 | 已有标准矩阵数据和成本模型 |
| MLflow | 实验和模型管理 | 实验数量增多，需要对比和注册 |
| PostgreSQL | 元数据和交易记录 | 多任务、多用户或纸面交易开始 |
| Prefect | 调度编排 | 数据任务超过 5 个且需要依赖管理 |
| vn.py | 交易架构参考或 gateway | 目标券商接口匹配 |
| QMT/PTrade | 实盘适配 | 已获得接口权限并通过沙盒/仿真验证 |

## 6. 二次落地计划

### 6.1 里程碑 A：数据可信闭环

周期：4-6 周。

目标：

- 建立 A 股日频数据的采集、清洗、复权和校验流程。
- 支持可复现地读取任意日期的股票池和价格数据。

范围：

- 股票主数据。
- 交易日历。
- 日线行情。
- 复权因子。
- 停牌状态。
- 涨跌停价格。
- 指数成分历史，先选一个指数池。

验收标准：

- 任意交易日可生成股票池。
- 数据缺失率、重复率、异常价格有报告。
- 至少两个数据源抽样对账。
- 回测读取固定数据版本。

暂不做：

- 分钟数据。
- 财务数据深度处理。
- 新闻公告 NLP。
- 实时行情。

### 6.2 里程碑 B：单因子研究闭环

周期：3-4 周。

目标：

- 建立标准因子评价流程。
- 证明系统能稳定评估一个简单因子。

范围：

- 因子：动量、反转、波动率、流动性、市值。
- 处理：去极值、标准化、行业中性化可后置，第一版至少做标准化。
- 评价：IC、Rank IC、分层收益、多空收益、换手。

验收标准：

- 每个因子生成一份固定格式报告。
- 报告包含样本内、样本外或滚动窗口表现。
- 能发现明显异常，例如未来函数导致的异常高 IC。

暂不做：

- LightGBM。
- 深度学习。
- 自动因子挖掘。

### 6.3 里程碑 C：日频组合回测闭环

周期：3-5 周。

目标：

- 将因子信号转成持仓组合，并生成可解释回测结果。

范围：

- 指数成分池内选股。
- 周频或月频调仓。
- 等权持仓。
- 成本模型。
- 停牌和涨跌停保守处理。
- 买入 100 股取整。
- T+1 卖出限制。

验收标准：

- 输出净值、超额收益、最大回撤、年化收益、波动率、夏普、换手。
- 输出每期持仓和每笔交易。
- 输出不可成交订单统计。
- 与基准指数对比。

暂不做：

- 多策略资金分配。
- 复杂组合优化。
- 分钟级执行。

### 6.4 里程碑 D：事件回测最小可用版

周期：6-10 周。

目标：

- 将组合目标转成订单、成交、持仓和现金流水。
- 为纸面交易和实盘执行打基础。

范围：

- 事件：MarketData、Signal、Order、Trade、Position、Account、RiskEvent。
- 模拟券商：处理下单、成交、撤单、资金、持仓。
- 撮合：日频开盘价/VWAP/收盘价三种模式。
- 风控：单票权重、单笔金额、总仓位、现金约束。

验收标准：

- 每笔订单状态可追踪。
- 账户现金和持仓可复算。
- 风控拦截有日志。
- 同一策略与日频组合回测差异可解释。

### 6.5 里程碑 E：纸面交易

周期：8-12 周。

目标：

- 用真实当日数据生成信号和模拟订单。
- 不接真实交易前，验证运行稳定性。

范围：

- 每日数据更新。
- 盘前检查。
- 盘后生成信号。
- 次日模拟成交。
- 日报和异常告警。

验收标准：

- 连续 20 个交易日稳定运行。
- 每天有数据检查报告、交易模拟报告、持仓报告。
- 异常时能停止并给出原因。

### 6.6 里程碑 F：小资金实盘预研

周期：不固定，至少 2-3 个月。

目标：

- 明确券商接口可行性。
- 做沙盒或小资金人工确认式交易。

范围：

- 调研 QMT、PTrade 或目标券商 API。
- 定义 BrokerAdapter。
- 做账户查询、持仓查询、订单查询、下单、撤单最小闭环。
- 先人工确认下单，再逐步自动化。

准入条件：

- 纸面交易通过。
- 风控规则通过演练。
- 对账流程通过。
- 人工暂停和撤单可用。

## 7. 建议修订原文的重点

建议将原文改成两层结构：

- 将阶段时间改成更保守估计。

## 8. 最终建议

最稳的启动方式不是先搭大平台，而是先完成一个小而硬的闭环：

```text
日频数据可信
  -> 单因子评价可信
  -> 多因子组合回测可信
  -> A 股规则事件回测可信
  -> 纸面交易稳定
  -> 小资金实盘
```

第一版系统的价值判断标准也应简单：

- 同一天、同配置、同数据版本，能跑出完全一致的结果。
- 回测中的每笔交易都能解释为什么买、为什么卖、为什么没成交。
- 因子收益不是由未来函数、幸存者偏差或错误复权造成的。
- 交易成本、换手和不可成交会被清楚展示。
- 实盘前系统已经能在纸面交易中稳定运行。

如果这些基础没有打牢，接入再高级的模型或交易框架也只会放大错误；如果这些基础打牢，后续引入 Qlib、vectorbt、MLflow、vn.py 或券商接口都会顺很多。
