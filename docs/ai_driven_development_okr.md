# AI 驱动的国内股票量化系统开发路径 OKR

版本：v0.1  
日期：2026-05-17  
适用方向：国内 A 股股票，优先日频和低频多因子策略，逐步扩展到事件回测、纸面交易和小资金实盘。

## 1. 路线原则

本路线基于以下两份文档收敛：

- `docs/domestic_stock_quant_system.md`
- `docs/domestic_stock_quant_system_review_plan.md`

由于系统主要依靠 AI 持续迭代，开发路径不以固定周期为核心，而以“阶段目标、关键结果、验收证据、退出标准”为核心。每一阶段只在通过关键结果后进入下一阶段，避免在数据和回测还不可靠时过早进入模型、实盘或复杂平台化。

总路线：

```text
工程骨架可信
  -> 数据可信
  -> 因子研究可信
  -> 组合回测可信
  -> 事件回测可信
  -> 纸面交易可信
  -> 小资金实盘可控
  -> 多策略平台化
```

核心原则：

- 先可靠，再复杂。
- 先日频，再分钟级。
- 先研究闭环，再交易闭环。
- 先纸面交易，再真实资金。
- 先可复现，再追求收益。
- 所有阶段都必须留下测试、日志、报告和可追溯产物。

## 2. 最终愿景

建设一个可长期迭代的国内股票量化交易系统，覆盖以下完整流程：

- 数据采集、清洗、校验、版本管理。
- 股票池构建、因子计算、因子评价。
- 多因子组合构建和保守回测。
- A 股规则事件驱动回测。
- 纸面交易、实盘前验证和风控演练。
- 券商接口适配、小资金实盘。
- 盘后归因、绩效复盘、策略生命周期管理。
- 多策略、多账户、多组合的平台化管理。

最终系统不追求一开始“大而全”，而是每一层都能独立验证、替换和回滚。

## 3. 技术路线分层

### 3.1 第一阶段核心技术栈

第一阶段只保留必要工具：

| 模块 | 选型 |
|---|---|
| 语言 | Python 3.11+ |
| 项目管理 | uv |
| 配置 | Pydantic + YAML/TOML |
| 数据格式 | Parquet |
| 本地查询 | DuckDB |
| 数据处理 | pandas、polars、pyarrow |
| 测试 | pytest |
| 报告 | Markdown/HTML/CSV/JSON |
| 日志 | structlog 或标准 logging |
| 回测 | 自研日频回测核心 |

第一阶段暂不把 Qlib、vn.py、ClickHouse、Kubernetes、完整 OMS/EMS、Grafana 作为核心依赖。

### 3.2 后续候选技术

| 能力 | 候选工具 | 引入条件 |
|---|---|---|
| 高速参数扫描 | vectorbt | 已有标准矩阵数据和成本模型 |
| AI/ML 因子研究 | Qlib、LightGBM | 单因子研究和基础回测稳定 |
| 实验追踪 | MLflow | 策略和实验数量明显增加 |
| 调度 | Prefect 或 Airflow | 数据任务依赖复杂，手工运行不可控 |
| 分析数据库 | ClickHouse | 分钟级数据或多人研究场景出现 |
| 实盘框架参考 | vn.py | 目标券商接口匹配或架构复用明确 |
| 交易接口 | QMT、PTrade、券商 API | 纸面交易稳定且接口权限明确 |
| 监控 | Prometheus、Grafana | 进入纸面交易后逐步引入 |

## 4. AI 迭代工作方式

### 4.1 每轮 AI 任务的标准格式

每次让 AI 开发时，任务应尽量收敛成一个可验证工作包：

```text
目标：
  本轮要完成什么用户价值或系统能力。

范围：
  本轮修改哪些模块，不修改哪些模块。

输入：
  需要的数据、配置、接口或文档。

输出：
  代码、测试、报告、文档或样例。

验收：
  运行哪些命令，看到哪些结果算通过。

约束：
  不引入哪些依赖，不改变哪些接口，不做哪些扩展。
```

### 4.2 每轮 AI 交付物要求

每个任务默认需要交付：

- 实现代码。
- 最小测试。
- 示例配置或样例数据。
- 文档更新。
- 可运行命令。
- 已知限制说明。

如果任务只做调研或设计，也必须交付：

- 决策结论。
- 替代方案。
- 风险点。
- 下一步可执行任务。

### 4.3 每轮 AI 验收规则

所有迭代都遵循：

- 能用测试证明的，不只靠解释。
- 能用样例跑通的，不只写接口。
- 能固定输入输出的，不依赖隐式状态。
- 能记录版本的，不覆盖历史结果。
- 涉及资金、订单和风控的代码必须有明确失败路径。

### 4.4 任务拆分粒度

推荐每个 AI 任务只做一种变化：

- 新增一个数据模型。
- 新增一个数据采集器。
- 新增一个数据校验规则。
- 新增一个因子。
- 新增一个回测成本模型。
- 新增一类报告。
- 修复一个明确 bug。

不推荐单轮任务：

- “实现完整量化系统”。
- “接入所有数据源”。
- “做一个全功能回测引擎”。
- “直接实盘交易”。

## 5. OKR 总览

| 阶段 | Objective | 关键门槛 |
|---|---|---|
| O0 | 建立工程基座 | 项目可安装、测试可运行、核心类型稳定 |
| O1 | 建立可信数据闭环 | 日频数据可采集、校验、版本化、复现 |
| O2 | 建立因子研究闭环 | 单因子评价报告可信，避免未来函数 |
| O3 | 建立组合回测闭环 | 多因子策略能生成持仓、交易和绩效报告 |
| O4 | 建立 A 股事件回测闭环 | 订单、成交、现金、持仓可追踪 |
| O5 | 建立纸面交易闭环 | 每日自动运行、模拟交易、日报和异常处理 |
| O6 | 建立实盘预研闭环 | 券商接口、风控、对账、人工接管路径明确 |
| O7 | 建立小资金实盘闭环 | 小资金、白名单策略、可暂停、可对账 |
| O8 | 建立平台化扩展能力 | 多策略、多组合、实验追踪、监控和调度 |

## 6. O0：工程基座可信

### Objective

建立一个可以长期由 AI 迭代的 Python 工程骨架，保证项目结构清晰、依赖可控、测试可运行、核心领域模型稳定。

### Key Results

- KR0.1：项目可以通过统一命令安装和运行测试。
- KR0.2：核心目录结构形成，包括 `qtrade/`、`tests/`、`configs/`、`docs/`。
- KR0.3：定义核心领域模型：`Symbol`、`Bar`、`Signal`、`TargetPosition`、`Order`、`Trade`、`Position`、`Account`。
- KR0.4：配置系统支持环境区分，例如 `dev`、`research`、`paper`。
- KR0.5：基础日志和错误类型可用。

### 手段路径

- 使用 `uv` 初始化项目。
- 使用 `Pydantic` 定义配置和核心数据结构。
- 使用 `pytest` 建立基础测试。
- 使用 `ruff` 或同类工具保证代码风格。
- 建立 `examples/` 放最小运行样例。

### AI 任务拆分

- 初始化 `pyproject.toml` 和包结构。
- 定义核心领域模型。
- 定义配置加载器。
- 定义日志初始化。
- 添加基础单元测试。
- 写入开发说明文档。

### 验收证据

- `pytest` 通过。
- 可以加载一份样例配置。
- 可以创建一条 `Bar`、一条 `Signal` 和一条 `Order`。
- 文档说明如何运行测试和样例。

### 退出标准

当核心模型稳定到可以被数据、研究和回测模块复用时，进入 O1。

## 7. O1：数据可信闭环

### Objective

建立 A 股日频数据的采集、清洗、校验和版本化流程，保证后续研究和回测使用的数据可复现、可检查、可追溯。

### Key Results

- KR1.1：支持交易日历、股票主数据、日线行情、复权因子、停牌状态、涨跌停价格。
- KR1.2：数据按 `raw/clean/feature` 分层存储。
- KR1.3：所有数据带有 `source`、`as_of_date`、`ingested_at`、`version`。
- KR1.4：可以读取任意交易日的可交易股票池。
- KR1.5：每日数据质量报告包含缺失率、重复率、异常价格、交易日覆盖率。
- KR1.6：至少支持一个开发数据源，并保留第二数据源抽样对账接口。

### 手段路径

- 先接一个免费数据源做开发验证，例如 AkShare、BaoStock 或 Tushare。
- 数据落地使用 Parquet 分区。
- 使用 DuckDB 做本地查询和抽样检查。
- 数据校验规则先写成独立模块，不嵌入采集器。
- 数据源接口采用 `DataProvider` 抽象，避免后续切换成本过高。

### AI 任务拆分

- 定义数据表 schema。
- 实现交易日历采集。
- 实现股票主数据采集。
- 实现日线行情采集。
- 实现复权因子采集。
- 实现停牌和涨跌停数据处理。
- 实现 Parquet 存储和读取。
- 实现数据质量检查。
- 生成数据质量报告。

### 验收证据

- 可以生成指定日期股票池。
- 可以读取某只股票一段时间的日线数据。
- 数据质量报告能指出缺失、重复、异常价格。
- 同一数据版本重复读取结果一致。

### 暂不做

- 分钟线。
- Tick 数据。
- 新闻公告 NLP。
- 多数据源自动融合。
- 实时行情。

### 退出标准

当日频研究所需数据可以稳定生成，并且质量报告能发现明显异常时，进入 O2。

## 8. O2：因子研究可信

### Objective

建立单因子和基础多因子的研究流程，保证因子评价避免未来函数、幸存者偏差和明显数据错误。

### Key Results

- KR2.1：支持基础股票池过滤：ST、停牌、上市不足 N 日、低流动性、退市风险。
- KR2.2：实现至少 5 个透明因子：动量、反转、波动率、流动性、市值。
- KR2.3：支持因子标准化、缺失处理、去极值。
- KR2.4：输出 IC、Rank IC、ICIR、分层收益、换手、覆盖率。
- KR2.5：报告区分训练区间、验证区间和测试区间，或支持滚动样本外评价。
- KR2.6：因子结果可保存到 `feature` 层并带版本。

### 手段路径

- 不急于引入 ML 模型。
- 先做透明、可解释、可复算的规则因子。
- 所有因子计算都必须声明使用的数据时间点。
- 财务类因子暂缓，直到可得日和公告日处理可靠。
- 每个因子单独生成报告。

### AI 任务拆分

- 实现股票池过滤器。
- 实现动量因子。
- 实现反转因子。
- 实现波动率因子。
- 实现流动性因子。
- 实现市值因子。
- 实现因子清洗流程。
- 实现 IC 分析。
- 实现分层收益分析。
- 实现因子报告模板。

### 验收证据

- 每个因子都有独立报告。
- 因子报告包含覆盖率和缺失率。
- 未来收益标签和因子日期严格错开。
- 异常高 IC 会被标记为可疑。

### 暂不做

- LightGBM。
- 深度学习。
- 自动因子挖掘。
- 财务公告 NLP。
- 高频因子。

### 退出标准

当基础因子评价流程稳定，并能解释每个因子的有效性和风险时，进入 O3。

## 9. O3：组合回测可信

### Objective

把因子信号转成组合持仓和交易记录，建立保守、可解释、可复现的日频组合回测闭环。

### Key Results

- KR3.1：支持从因子分数生成目标持仓。
- KR3.2：支持周频、月频、指定日期调仓。
- KR3.3：支持等权、分数加权、权重上限。
- KR3.4：支持交易成本、滑点、印花税、佣金、过户费。
- KR3.5：支持停牌不可交易、涨停不可买、跌停不可卖。
- KR3.6：支持买入 100 股取整和现金约束。
- KR3.7：输出净值、超额收益、最大回撤、年化收益、波动率、夏普、换手、持仓、交易明细、不可成交统计。

### 手段路径

- 先做日频保守回测，不追求复杂撮合。
- 以指数成分池为第一批股票池，例如沪深 300 或中证 500。
- 使用固定基准对比策略表现。
- 所有交易明细都可追踪到信号来源。

### AI 任务拆分

- 定义 `PortfolioBuilder`。
- 定义调仓日历。
- 实现目标权重生成。
- 实现交易成本模型。
- 实现不可交易规则。
- 实现持仓和现金更新。
- 实现绩效指标。
- 实现回测报告。
- 实现策略样例。

### 验收证据

- 一个多因子样例可以完整跑通。
- 报告包含绩效、交易、持仓和不可成交。
- 同一配置重复运行结果一致。
- 所有收益都能追踪到持仓变化。

### 暂不做

- 分钟级回测。
- Tick 撮合。
- 复杂组合优化器。
- 多策略资金分配。

### 退出标准

当日频组合回测结果可解释，并能稳定生成完整报告时，进入 O4。

## 10. O4：A 股事件回测可信

### Objective

建立最小可用的事件驱动回测系统，让订单、成交、现金、持仓、风控事件都可追踪，为纸面交易和实盘执行打基础。

### Key Results

- KR4.1：定义事件模型：`MarketDataEvent`、`SignalEvent`、`OrderEvent`、`TradeEvent`、`RiskEvent`、`AccountEvent`。
- KR4.2：支持模拟券商处理订单、成交、撤单、持仓和现金。
- KR4.3：支持 T+1 卖出限制。
- KR4.4：支持日频开盘价、收盘价或 VWAP 三种成交模式。
- KR4.5：支持订单状态机：新建、已提交、部分成交、全部成交、已撤销、被拒绝。
- KR4.6：支持基础风控：单票权重、单笔金额、总仓位、现金约束。
- KR4.7：事件回测与组合回测差异可解释。

### 手段路径

- 事件回测先服务于日频策略，不做高频。
- 先实现保守规则，再逐步增加撮合细节。
- 风控引擎先做硬规则，不做复杂风险模型。
- 所有事件都写入日志或结果表。

### AI 任务拆分

- 定义事件基类和事件总线。
- 实现订单模型和状态机。
- 实现模拟券商。
- 实现 T+1 持仓锁定。
- 实现撮合器。
- 实现基础风控规则。
- 实现事件回测报告。
- 实现与 O3 回测的结果对比工具。

### 验收证据

- 每笔订单都有状态流转。
- 每笔成交会影响现金和持仓。
- 风控拦截有原因。
- 组合回测和事件回测差异能被报告解释。

### 暂不做

- 真实盘口队列。
- 复杂部分成交模型。
- 融资融券。
- ETF 申赎。
- 分钟级执行算法。

### 退出标准

当事件回测能稳定模拟一套日频策略的完整交易过程，并能解释与组合回测的差异时，进入 O5。

## 11. O5：纸面交易可信

### Objective

建立不接真实资金的每日运行系统，用真实或准实时数据生成信号、模拟订单、模拟成交、持仓和日报，验证系统运行稳定性。

### Key Results

- KR5.1：支持每日数据更新和盘前检查。
- KR5.2：支持定时生成信号和目标持仓。
- KR5.3：支持 PaperBroker 模拟订单和成交。
- KR5.4：支持每日持仓、订单、成交、风控、异常日报。
- KR5.5：支持异常中止，例如数据缺失、行情延迟、账户不一致。
- KR5.6：支持最小告警机制，例如日志、邮件、企业微信或本地报告。
- KR5.7：连续多个交易日运行时，结果可追踪、可复盘。

### 手段路径

- 先用 CLI 和本地报告，不急着做 UI。
- 每日任务拆成盘前、盘中、盘后三类检查。
- 纸面交易使用和事件回测相同的订单、成交、账户模型。
- 建立“今日是否允许交易”的准入检查。

### AI 任务拆分

- 实现运行日历和任务入口。
- 实现盘前数据检查。
- 实现每日信号生成任务。
- 实现 PaperBroker。
- 实现每日交易报告。
- 实现异常处理和中止机制。
- 实现简单告警。

### 验收证据

- 可以连续运行多个模拟交易日。
- 每日有报告。
- 数据缺失时系统停止而不是继续交易。
- 纸面交易结果能被事件回测复盘。

### 暂不做

- 真实下单。
- 自动接管券商账户。
- 复杂 Web 前端。
- 多账户管理。

### 退出标准

当纸面交易稳定、异常可解释、日报可复盘，并且人工能理解系统每天做了什么时，进入 O6。

## 12. O6：实盘预研可信

### Objective

在不贸然自动交易的前提下，验证券商接口、权限、订单路径、对账路径和人工接管流程，为小资金实盘做准备。

### Key Results

- KR6.1：确定目标券商和接口方案，例如 QMT、PTrade 或券商专用 API。
- KR6.2：定义并实现 `BrokerAdapter` 最小接口。
- KR6.3：支持账户查询、持仓查询、订单查询、成交查询。
- KR6.4：支持沙盒或测试环境下单和撤单；若无沙盒，先做人工确认模式。
- KR6.5：支持盘前、盘中、盘后对账。
- KR6.6：支持手动暂停、一键撤单、禁止开仓。
- KR6.7：形成实盘准入清单。

### 手段路径

- 实盘接口独立于研究和回测模块。
- 券商适配器只实现标准接口，不让策略直接调用券商 API。
- 初期采用人工确认下单，减少自动化风险。
- 所有真实账户相关操作必须有日志。

### AI 任务拆分

- 定义 `BrokerAdapter` 协议。
- 实现 MockBrokerAdapter。
- 根据目标券商实现查询接口。
- 实现订单提交的人工确认模式。
- 实现对账报告。
- 实现实盘前检查清单。
- 实现紧急暂停状态。

### 验收证据

- 可以读取真实或模拟账户信息。
- 可以生成对账报告。
- 下单前有风控检查和人工确认。
- 暂停状态下任何新订单都会被拒绝。

### 暂不做

- 无人值守自动实盘。
- 多券商同时接入。
- 高并发订单执行。
- 高频交易。

### 退出标准

当券商接口路径清楚，风控、对账、人工接管都通过演练时，进入 O7。

## 13. O7：小资金实盘可控

### Objective

以白名单策略、白名单股票池和固定资金上限进行小资金实盘，验证从信号到真实成交再到盘后复盘的完整闭环。

### Key Results

- KR7.1：只允许通过审核的策略上线。
- KR7.2：只允许白名单股票池和资金上限。
- KR7.3：所有订单必须经过风控。
- KR7.4：每日盘前检查、盘中监控、盘后对账必须完成。
- KR7.5：实盘成交与系统持仓一致。
- KR7.6：异常时可以暂停、撤单、禁止开仓。
- KR7.7：实盘表现能与纸面交易和回测对比。

### 手段路径

- 小资金阶段优先低频策略。
- 每次调仓前人工复核目标订单。
- 每日盘后必须生成实盘复盘报告。
- 任何异常都优先暂停，而不是自动修复。

### AI 任务拆分

- 实现实盘运行模式。
- 实现白名单策略和股票池。
- 实现资金上限检查。
- 实现实盘日报。
- 实现回测、纸面、实盘三方对比。
- 实现异常处理手册。

### 验收证据

- 真实订单、成交、持仓、资金可对账。
- 实盘日报可解释每笔交易。
- 风控拦截有记录。
- 人工暂停和恢复流程可执行。

### 暂不做

- 大资金。
- 多策略自动分配。
- 全自动无人监管。
- 日内高频策略。

### 退出标准

当小资金实盘稳定，并且交易、风控、对账、复盘形成制度化流程时，进入 O8。

## 14. O8：平台化扩展

### Objective

从单策略、单账户、低频系统扩展成多策略、多组合、可监控、可治理的量化平台。

### Key Results

- KR8.1：支持多策略并行研究和回测。
- KR8.2：支持策略版本、参数版本、数据版本和结果版本追踪。
- KR8.3：引入 MLflow 或等价系统管理实验。
- KR8.4：引入 Prefect/Airflow 管理数据和研究任务。
- KR8.5：引入 Prometheus/Grafana 管理运行监控。
- KR8.6：支持策略上线、暂停、下线的审批流程。
- KR8.7：支持组合层资金分配和策略相关性分析。

### 手段路径

- 先把单策略流程产品化，再做多策略。
- 先标准化报告和元数据，再引入平台工具。
- 监控优先覆盖数据任务、交易任务、账户状态和风控事件。
- 策略生命周期管理必须包含上线前、运行中和下线后复盘。

### AI 任务拆分

- 实现实验记录模块。
- 接入 MLflow。
- 实现任务调度 DAG。
- 实现监控指标导出。
- 实现多策略配置。
- 实现组合资金分配。
- 实现策略审批文档模板。

### 验收证据

- 多个策略可以独立回测和比较。
- 每个策略版本可追踪数据、参数和结果。
- 系统运行状态可以被监控。
- 策略上线和下线有记录。

## 15. 阶段推进规则

### 15.1 不允许跳阶段的事项

- 数据没有质量报告，不进入因子研究。
- 因子没有样本外评价，不进入组合实盘讨论。
- 组合回测没有交易明细，不进入事件回测。
- 事件回测没有订单状态，不进入纸面交易。
- 纸面交易不稳定，不进入真实资金。
- 没有风控和对账，不允许自动下单。

### 15.2 可以并行的事项

- 文档和代码样例可以并行。
- 数据 schema 和核心类型可以并行。
- 报告模板和指标计算可以并行。
- MockBroker 和 BrokerAdapter 接口设计可以早于真实券商接入。

### 15.3 必须沉淀的资产

每个阶段结束时，都要沉淀：

- 可运行命令。
- 测试用例。
- 样例配置。
- 样例数据或数据生成器。
- 报告样例。
- 已知限制。
- 下一阶段任务列表。

## 16. 关键质量门槛

### 16.1 数据质量门槛

- 交易日历完整。
- 股票主数据可复现历史状态。
- 行情数据无明显重复和断档。
- 复权数据有版本。
- 停牌和涨跌停可被回测使用。
- 财务数据未处理公告日前，不用于实盘策略。

### 16.2 回测质量门槛

- 不允许未来函数。
- 不允许使用当前指数成分回测历史。
- 不允许忽略停牌、涨跌停和 T+1。
- 不允许只给净值曲线而没有交易明细。
- 不允许没有交易成本。
- 不允许没有基准对比。

### 16.3 实盘质量门槛

- 所有订单经过风控。
- 订单、成交、持仓、资金可对账。
- 异常时系统优先停止。
- 人工可以暂停、撤单、禁止开仓。
- 每日有盘前检查和盘后复盘。
- 实盘策略必须能追溯到回测版本和纸面交易记录。
